System Management via API

Get System Types

The HTTP method to be used is:
Request URI will be /systemtype
Parameters: None


Get all Systems

The HTTP method to be used is:
Request URI will be /system
Parameters: fields fields
Comma separated list of the field names that you would like to be returned. If not specified, all fields are returned.
Default: All fields

Request URI will be /system

For this group of API requests, the various possibilities are listed on the left. Each one is a form which you can complete and then click on the blue submit button. When you click on a button for the first time, results will be in a new tab or window. Once the second tab/window has been created, every time you click on a button the results will be in that tab/window. This page should remain until you choose to close it.

The API is capable of replying with JSON responses, but this page is configured to ask the API to answer with HTML. The HTML sent by the API is actually JSON, but prettified and presented as HTML so as to be readable in a browser.

Results from the API

The API can provide results as either HTML or JSON. This can be controlled in two ways. One way is to send, with the request, an "Accept" header specifying either "application/json" or "text/html". If neither is specified, the default is HTML. The other way is to suffix the request URI with either .json or .html.

Where a request to the API can result in multiple answers, for example if a request is made for all nodes in a system, the result will always be an array. This will be so even if there is only one node, or if there are no nodes at all.

In general, where a particular type of entity is expected from the API, and the result is null, the API will return something of the type that would be expected if the result were not null. So if the result could be an object, but is null, the API will return a null object. If the result could be a string, the API will return an empty string. Null is only used for cases such as numbers, where there is no naturally null value.

Many requests will accept a "fields" parameter, whose value is a comma separated list of field names. The API will return only the fields in the list in the results, thus enabling selective retrieval of only the data required. The efficiency of the API may be significantly improved if requests are restricted to the data that is needed.

Date/time stamps are always returned by the API in RFC 2822 date format, which allows them to be reliably interpreted regardless of differing locales.

Parameters to API calls

The form fields that are needed to make the Request URI are usually mandatory. If they are omitted, the request will still be sent to the API but results will vary according to the exact Request URI sent and may be erroneous.

By and large, parameters are optional. If they are not, you will see an error return when they are omitted. The Request URI is the information that will be added to the location to which requests are being sent (shown above). It is literal text except for items that are enclosed in angle brackets. Items in angle brackets are taken from the form field with a similar name. There is one exception to this rule: <daterange> is not part of a resource and so does not appear in any form. It should consist of an optional date, a comma and another optional date. The two dates are from and to respectively. They should be in a format understood by the PHP strtotime() function, which only understands English.

When using this HTML explorer, optional parameter fields will have a pink background after you put the cursor into them. If a field has a pink background, it will be sent to the API, even if it is empty. Where an optional field has a white background, it will not be sent to the API. You can clear the pink background by double clicking in the field.

If parameters (apart from the information in the URI) are submitted with a GET request, they must be supplied in the form of a standard query string. That is, a series of name-data assignments separated by ampersand. For example, 'systemid=4&nodeid=2'. Where a GET request is for a whole set of Tasks, Schedules or Backups any of the fields for the entity involved can be used as filtering parameters, although this cannot always be achieved via this tool.

Any request can use the "suppress_response_codes" parameter with a value of "true". If this is set, then every request will receive an HTTP return code of 200 OK, but the actual code is included in the response as a property called "httpcode".

Where parameters are submitted with an HTTP PUT request, there is a single string passed with the request. The string should be a query string, similar to that provided for GET.

Where parameters are submitted with an HTTP POST request, the usual conventions for POST apply, so that each parameter is a POST data item.

This page does not check information submitted in the form fields, but sends it directly to the API. If the information submitted is invalid, the API will send back an error message.
